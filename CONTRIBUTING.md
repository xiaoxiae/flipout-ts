# Contributing to flipout-ts

Thanks for taking a look. The repo is small and the porting conventions are
strict — please skim this before opening a PR.

## Project structure

```
flipout-ts/
├── src/             # library source (published as `flipout-ts`)
│   ├── math/        # L0 — Vec2/Vec3, barycentric, predicates
│   ├── mesh/        # L1 — half-edge SurfaceMesh, iteration, flips
│   ├── geometry/    # L2 — VertexPositionGeometry (extrinsic)
│   ├── intrinsic/   # L3 — SignpostIntrinsicTriangulation
│   ├── flipout/     # L4 — FlipEdgeNetwork, the algorithm
│   └── three/       # L5 — Three.js adapter (only place that imports `three`)
├── test/unit/       # vitest specs, mirrors src/ layer layout
├── fixtures/        # golden JSON regenerated by tools/gen_fixtures.py
├── tools/           # Python regenerator scripts (not shipped)
└── demo/            # standalone Vite app — interactive Three.js demo
```

## Build / test

```bash
npm install
npm run test        # 688 tests, must stay green
npm run typecheck   # tsc --noEmit, must be clean
npm run build       # emits dist/

cd demo
npm install
npm run build
npx tsx scripts/smoke.mts   # node-level end-to-end import check
```

## Porting convention

Each source file under `src/` carries a header banner naming the
geometry-central source file (and ideally the SHA) it was ported from. New
files keep the same convention — even if the C++ side is just a single class
or namespace, point to it. The barrel `index.ts` files (one per layer) cite
the corresponding geometry-central namespace.

`src/three/` is the only module without a geometry-central counterpart;
geometry-central has no Three.js dependency, so this is new code.

## Test discipline

- 688 tests must stay green. `main` has zero skipped tests by design.
- New code adds tests. Three styles are accepted (see `README.md` →
  *Test strategy*): mathematical invariants, hand-computed examples, golden
  fixtures from `potpourri3d`.
- Regenerating fixtures requires Python (`tools/gen_fixtures.py`) — but the
  TS suite reads the committed JSON directly, no Python at runtime.

## Algorithmic context

The algorithm is described in
[Sharp & Crane, *You Can Find Geodesic Paths in Triangle Meshes by Just
Flipping Edges*, SIGGRAPH Asia 2020](https://nmwsharp.com/media/papers/flip-geodesics/flip_geodesics.pdf).
Read this before changing anything in `src/flipout/` or
`src/intrinsic/`.

## Branches

- `main` — stable, all tests passing, what gets published.
- `bezier-experiment` — experimental Bezier subdivision on top of the
  geodesic. Has visual-quality issues at deep subdivision rounds; do *not*
  merge into `main` without a separate review.
